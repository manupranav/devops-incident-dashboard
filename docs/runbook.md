## Overview

This runbook covers operational responses for alerts generated by `monitor.py` and dispatched via `alert.sh`. Follow these procedures in order. Do not skip steps.

**First rule of incident response:** Don't panic. Work the problem.

**Important:** Always check `logs/alerts.log` first to understand alert history before taking action.

## Alert Severity Levels

| Level    | Meaning                              | Response Time | Who Gets Paged |
|----------|--------------------------------------|---------------|----------------|
| WARNING  | Something is trending bad            | 30 minutes    | On-call (Slack) |
| CRITICAL | System at risk, user impact possible | 5 minutes     | On-call (PagerDuty + Slack) |

## CPU High Usage

### Symptoms
- Alert: `CPU WARNING` — usage above 75%
- Alert: `CPU CRITICAL` — usage above 90%

### Immediate Steps

```bash
# 1. Check what's consuming CPU
top -bn1 | head -20

# 2. Find top CPU processes
ps aux --sort=-%cpu | head -10

# 3. Check if a cron job is running
crontab -l

# 4. Check system load average
uptime
```

### Resolution
- If a rogue process: `kill -15 <PID>` (graceful) or `kill -9 <PID>` (force)
- If legitimate load spike: notify stakeholders, monitor for 15 minutes
- If CPU stays above 90% for >10 minutes: escalate to senior engineer

## Memory High Usage

### Symptoms
- Alert: `Memory WARNING` — usage above 80%
- Alert: `Memory CRITICAL` — usage above 90% (OOM kill risk)

### Immediate Steps

```bash
# 1. Check memory usage breakdown
free -h

# 2. Find memory-hungry processes
ps aux --sort=-%mem | head -10

# 3. Check for memory leaks in app processes
# (Replace 'myapp' with actual process name)
cat /proc/$(pgrep myapp)/status | grep -i vmrss

# 4. Check swap usage
swapon --show
```

### Resolution
- Restart leaking service if identified (coordinate with app team first)
- If OOM is imminent: `sync; echo 3 > /proc/sys/vm/drop_caches` (safe, frees page cache)
- If swap is full: immediate escalation required

## Disk Space Critical

### Symptoms
- Alert: `Disk WARNING` — usage above 75%
- Alert: `Disk CRITICAL` — usage above 90%

### Immediate Steps

```bash
# 1. Find which filesystem is full
df -h

# 2. Find largest directories
du -sh /* 2>/dev/null | sort -rh | head -20

# 3. Check for log file bloat (common culprit)
du -sh /var/log/* | sort -rh | head -10

# 4. Find files larger than 1GB
find / -type f -size +1G 2>/dev/null
```

### Resolution
- Rotate and compress old logs: `logrotate -f /etc/logrotate.conf`
- Clear package manager cache: `apt clean` or `yum clean all`
- Remove old Docker images: `docker system prune -f` (confirm with team first)
- If /tmp is full: `rm -rf /tmp/*` (safe only if confirmed no active temp usage)

## Monitor Process Down

### Symptoms
- No entries in `logs/monitor.log` for >15 minutes
- PagerDuty heartbeat check fails (TODO: heartbeat not yet implemented)

### Immediate Steps

```bash
# 1. Check if monitor.py is running
pgrep -fl monitor.py

# 2. Check last log entry
tail -50 logs/monitor.log

# 3. Check for Python errors
journalctl -u monitor --no-pager -n 50   # If running as systemd service

# 4. Restart monitor
python3 monitor.py &
```

## Escalation Policy

| Situation | First Contact | Escalate To | Escalate After |
|-----------|--------------|-------------|----------------|
| CPU/Memory WARNING | Slack #ops-alerts | On-call engineer | 30 min no response |
| CPU/Memory CRITICAL | PagerDuty | Senior engineer | 10 min no response |
| Disk CRITICAL | Slack #ops-alerts | On-call engineer | 15 min no response |
| Multiple CRITICAL simultaneously | PagerDuty + phone | Engineering manager | Immediately |

## Post-Incident Checklist

After resolving any CRITICAL incident:

- [ ] Document root cause in `logs/incidents/YYYY-MM-DD-<title>.md`
- [ ] Update this runbook if the issue revealed a gap
- [ ] Create a ticket for permanent fix (don't just patch and forget)
- [ ] Schedule a 30-min blameless post-mortem if user impact occurred
- [ ] Verify monitoring is still active after the fix
